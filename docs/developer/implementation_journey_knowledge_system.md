# Implementation Journey: The Knowledge System

This note records how Trellis' knowledge system changed from a set of useful
but partly isolated memories into a more shared substrate across the agent
workflow.

## The Early Problem

The early knowledge system was already valuable, but it was fragmented.

Different parts of the workflow could each "know" something without that
knowledge being visible to the rest of the platform:

- the builder had one view of retrieved lessons
- the critic had a narrower or differently formatted view
- the model validator often reasoned from a separate context
- prompt fallback paths could assemble cookbook and lesson context
  differently from the main build path
- task runs and traces could preserve only fragments of what an agent had
  actually seen

The result was inconsistency more than absence. The platform could know
something and still repeat the same mistake in another stage.

## The First Useful Layer: Canonical Knowledge Assets

Before shared memory became a real platform feature, Trellis already had
valuable canonical assets:

- feature taxonomy
- product decompositions
- cookbooks
- method requirements
- import registry
- lessons
- principles and traces

These lived mainly under `trellis/agent/knowledge/` and its canonical YAML
files. That was the right direction, but those assets still needed a common way
to flow through the platform.

## The Shared-Memory Shift

The important architectural move was to stop thinking of retrieval as "fetch
some lessons for this prompt" and instead think of it as:

> build one shared knowledge payload that multiple agents can consume

That led to:

- common retrieval and formatting in `knowledge/retrieval.py`
- structured cross-agent observations captured during builds
- shared builder, reviewer, and routing prompt views
- similar-product lessons entering decomposition
- platform requests carrying the compiled shared knowledge payload
- traces and reflection preserving agent observations instead of only boolean
  outcomes

## Why This Mattered

This change reduced a specific class of mistakes:

- builder and validator disagreeing because they saw different prior lessons
- fallback prompts drifting away from the main retrieval path
- task reruns forgetting why a previous build failed
- one stage rediscovering what another stage already learned

It did not magically make the platform smarter. What it did was make its memory
less lossy.

## Promotion Became Part Of The Runtime

Another important change was that knowledge promotion stopped being only an
offline curation exercise.

The runtime now captures and persists:

- observations from builder/critic/model-validator stages
- knowledge traces
- cookbook candidates
- lessons tied to source traces
- gap logs and blocker reports
- adapter freshness reviews and adoption artifacts, including the raw
  ``_fresh`` drift plus the resolved lifecycle stage
- basket supersession decisions for stale guidance cleanup, recorded as
  read-only trace data until the canonical lesson index is updated elsewhere

That made the knowledge system more operational. It is no longer just a static
library of YAML assets; it is also the memory surface generated by task
execution.

## What Remains Isolated

Even after the shared-memory upgrades, a few mechanisms remain partly outside
the fully unified substrate:

- `analytics_cookbook.py` is still a more standalone guidance source
- `test_resolution.py` still carries older heuristic diagnosis logic
- direct deterministic `Session` and `Pipeline` flows do not always synthesize
  the full semantic shared-knowledge context unless they go through compile/build

These are not fatal problems, but they are reminders that the knowledge system
is still being pulled toward coherence rather than having reached it fully.

## The Most Important Design Lesson

The best knowledge-system decision was not "store more text." It was:

- normalize retrieval
- normalize formatting
- attach knowledge to requests and traces
- preserve observations in a reusable form

The system became stronger when memory became more structured and more
cross-stage, not when prompts became larger.

## Why This Matters For Guarded Autonomy

The long-term goal is a guarded multi-agent "Library Developer" loop.

That goal depends on shared memory because autonomous improvement fails if:

- the planner does not know the validator's past objections
- the builder does not know similar-product lessons
- the rerun loop does not know which traces created which lesson
- the UI cannot show what the platform learned from a failed attempt

So the knowledge-system journey is really the story of turning Trellis from a
collection of smart steps into a platform that can remember enough to improve
coherently.
